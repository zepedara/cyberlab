# Style Guide — the anti-jumble charter

This is a large, **public** cyber-training repository. Organization is priority #1: it must stay
navigable as it grows to dozens of modules across two VMs. Every contribution — human or loop —
follows the rules below. `cyberlab_validate.py` enforces the machine-checkable ones; reviewers
enforce the rest.

## 1. Numbered-folder naming
- Modules live at `modules/lab-linux/NN-slug/README.md` and `modules/lab-windows/NN-slug/README.md`.
- `NN` = zero-padded 2-digit order (`01`, `02`, …). Order is the intended learning/build sequence.
- `slug` = short kebab-case name of the tool group (`disk-forensics`, `static-re`). Lowercase,
  digits and single dashes only.
- One `NN-slug` = **one coherent tool group**, never a single tool and never a grab-bag. If a group
  gets too big, split it into two numbered modules.
- Optional per-module `exercise/` subdir holds the sample artifact + any held-out answer material.

## 2. The fixed 12-section contract
Every module README is exactly this, in this order, with these exact H2 titles (see
`docs/MODULE_TEMPLATE.md`):

1. `# NN · <Tool Group> — LAB-LINUX|LAB-WINDOWS`  (H1 title line)
2. `## Overview (plain language)`
3. `## Tools covered`
4. `## Learning objectives`
5. `## Environment check`
6. `## Guided walkthrough`
7. `## Hands-on exercise`
8. `## SOC analyst perspective`
9. `## Attacker perspective`
10. `## Answer key`
11. `## MITRE ATT&CK & DFIR phase`
12. `## Sources`

No missing sections, no extra sections, no reordering. Sub-headings inside a section use `###`.

## 3. Basic → SOC → attacker (the teaching arc)
Every module must teach the same tool three ways, and each of these three sections must be
substantial (>= 200 characters — stubs are rejected):
- **Overview** explains the tools in plain, jargon-free language first.
- **SOC analyst perspective** shows how a defender uses them for detection/IR, tied to Security
  Onion and MITRE ATT&CK.
- **Attacker perspective** shows the offensive use and, crucially, the **artifacts it leaves** for a
  defender to find.

## 4. Every claim is cited
`## Sources` carries at least one authoritative URL/citation (SANS, REMnux docs, mandiant/flare-vm,
kali.org/tools, MITRE, vendor tool docs). Tool facts trace to the master catalog. Sourced or it
did not happen.

## 5. Tools must be real
Every tool in `## Tools covered` must appear in `catalog/Cyber_Lab_VM_Build_Plan.md`. No invented
tools. The catalog is the single authority for what the two VMs contain.

## 6. Samples: benign only, hashed, safe-origin
- Practice artifacts live in the module's `exercise/` dir.
- Each is declared with its **type**, a **sha256**, and a **safe-origin note** (benign / inert /
  synthetic / EICAR / no-egress). **NEVER commit live or weaponized malware.** Detonation, if any,
  is described as happening only inside a no-egress snapshot.

## 7. INDEX.md is the single source of truth
`INDEX.md` is auto-generated by `cyberlab_validate.build_index()` — do not hand-edit it. It is the
one place that lists coverage (per-VM module tables + the coverage matrix). The validator's G7 gate
requires the modules listed in INDEX.md to exactly match the module dirs on disk; regenerate it
whenever a module is added or removed.

## 8. Cross-links use relative paths
Link between docs with repo-relative paths (`../../catalog/…`, `docs/STYLE_GUIDE.md`) so links work
on GitHub and in local clones alike. No absolute or machine-specific paths.
